# Skill：功能完成后的全局收口 Review

## 触发场景
- 一个 feature 已经基本写完，需要从全局代码视角再审一遍
- 用户要求“不要继续打补丁，要站在全局角度看设计是否冗余”
- 准备提交 commit 之前，需要确认这轮新增代码已经收口

## 目标
- 找出这轮 feature 新引入的**重复状态、重复语义、重复入口**
- 找出临时过渡代码是否应该删除
- 判断当前实现是否值得做一次**小重构**再提交
- 补齐必要测试和文档，使功能以框架方式落地，而不是以 patch 方式落地

## 步骤
1. **圈定本轮变更范围**
   - 用 `git status --short` 和 `git diff -- <paths>` 明确这轮 feature 影响的文件。
   - 不要泛看全仓库；先聚焦本轮变更路径和调用链。

2. **从“单一真源”角度检查设计**
   - 问 3 个问题：
     - 同一份状态是否在两个地方各维护一份？
     - 同一份语义是否被两个层重复解释？
     - 同一份配置是否在多个模块各自推导？
   - 如果答案是“是”，优先收敛到一个中心 plan / state / source of truth。

3. **按模块边界检查是否漂移**
   - 对照 `AGENTS.md` 的边界：
     - `data` 不写训练循环逻辑
     - `training` 不解析数据语义
     - `pipeline` 只做装配，不承载业务策略
     - 不重新引入与 CLI 平行的训练入口；独立控制面或可视化界面不进入 Shaft 核心
   - 发现某个 feature 在错误层落逻辑时，不要继续补丁，直接把它移回正确层。

4. **识别临时桥接代码并决定去留**
   - 常见信号：
     - callback 只是为了绕开真正应该放在 sampler / template / plan 层的逻辑
     - dataset、trainer、builder 同时维护相同的推导结果
     - 旧入口保留但已经没有必要
   - 如果桥接代码只是在过渡期存在，且现在已有更正确的落点，就删掉，不保留“双轨实现”。

5. **检查命名与风格**
   - 新类/函数/文件是否符合仓库主体命名：
     - 框架级能力：`Shaft*`
     - 模型专用实现：显式模型族名
   - 是否出现“看起来通用，实际只服务一个 feature”的泛名。
   - 是否出现临时变量/类型名把调用链搞混。

6. **判断是否需要小重构**
   - 满足以下任一条件，就先小重构再提交：
     - 有重复状态源
     - 有错误层承载逻辑
     - 有临时桥接代码已经暴露结构问题
     - 文档很难解释当前实现，因为实现本身绕
   - 小重构的目标是**减层、减重复、收真源**，不是扩功能。

7. **测试收口**
   - 至少补两类测试：
     - 直接覆盖本轮 bug / 冗余点的单测
     - 覆盖主链装配是否真的吃到新结构的集成测试
   - 如果重构涉及 trainer / dataloader / checkpoint / infer 这类主链装配，至少再跑一轮更接近真实调用链的 focused 回归。

8. **文档收口**
   - 行为或结构变化，至少同步一个正式文档：
     - `docs/architecture.md`
     - `docs/module_reference.md`
     - `docs/config_reference.md`
     - `docs/extension_guide.md`
     - `docs/online_eval_design.md`
   - 如果有明确延期的设计想法，不要半实现，写进 `docs/todo.md`。

## 输出要求
- review 结论优先写 **findings**，不要先写总结。
- 按严重性排序：
  - 阻塞级漏洞
  - 结构冗余
  - 命名/风格问题
  - 文档/测试缺口
- 如果已经做了小重构，说明：
  - 删掉了什么冗余
  - 真源现在收敛到哪里
  - 为什么这版比之前更干净

## 验收标准
- 这轮 feature 的核心状态或策略有**单一真源**
- 没有明显的临时桥接代码继续留在主链
- pipeline / trainer / data / model 的边界仍然清楚，且没有重新引入第二训练入口
- 新测试能锁住本轮发现的结构性问题
- 文档可以直接解释当前实现，而不是绕着解释 patch
