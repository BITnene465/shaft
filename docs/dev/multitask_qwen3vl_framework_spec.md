# Multi-Task Qwen3-VL Structured Training Framework Spec

> Draft v0.2
>
> This document is a normative implementation spec for agents.
> It defines the target behavior for the future framework.
> If code, tests, and this document disagree, this document wins.
>
> Use `MUST` / `SHOULD` / `MAY` in the RFC 2119 sense.

## 1. Goal

The framework MUST support training one shared Qwen3-VL weight across many structured vision tasks.

The initial target tasks are:

- `grounding` for arrow detection
- `keypoint_sequence` for arrow skeleton prediction

The framework MUST later scale to more structured detection tasks without splitting weights by task.

## 2. Non-Goals

This framework is not intended to:

- become a general-purpose LLM trainer for arbitrary text-only tasks
- hide task semantics inside the trainer
- infer task/domain from data implicitly
- rely on regex-based parsing inside the trainer
- couple deployment protocol with training internals

## 3. Agent Refactor Rules

Any agent modifying the training stack MUST obey the following rules:

1. Do not introduce a new semantic layer unless this document explicitly allows it.
2. Do not move task semantics into `core`.
3. Do not move sampling policy into `domains` or `tasks`.
4. Do not move domain codec logic into `core` or `tasks`.
5. Do not let prompt text become the source of truth for structured labels.
6. Do not change the batch routing rule without updating this spec first.
7. Do not use `eval_loss` as the sole best-model selector for multi-task structured training.

## 4. Core Design Principles

1. **One shared base model**
   - All tasks share one Qwen3-VL backbone.
   - Task differences are expressed through prompts, codecs, samplers, and metrics.

2. **Explicit routing**
   - Every sample MUST declare `task_type` and `domain_type`.
   - The framework MUST never guess the route.

3. **Codec-driven supervision**
   - Training targets MUST be generated by task codecs.
   - The trainer MUST consume structured metadata, not raw JSON text hacks.

4. **Prompt and codec are paired**
   - A task prompt profile and a codec profile MUST be aligned.
   - Prompt text is not the source of truth; codec is.

5. **Task-aware evaluation**
   - Each task MUST have its own metric.
   - Model selection MUST be based on a declared primary validation metric.

6. **Deployment protocol reuse**
   - The same task semantics SHOULD be reusable by inference and deployment code.
   - Training-specific behavior MUST stay out of deployment code.

7. **Adapter type is orthogonal to task mixing**
   - Multi-task mixing and finetune adapter choice are separate concerns.
   - The framework MUST support:
     - standard LoRA
     - DoRA
     - full fine-tuning when needed
   - Adapter type changes parameterization and deployment compatibility, but not the task routing model.

## 5. Concept Model

### 5.1 Route

A **route** is the pair:

```text
route = task_type / domain_type
```

Examples:

- `grounding/arrow`
- `keypoint_sequence/arrow`
- `joint_structure/arrow`

### 5.2 Task

A **task** defines:

- output schema
- codec
- loss metadata
- evaluation metric
- generation constraints

### 5.3 Domain

A **domain** defines:

- label set
- geometric conventions
- canonical ordering
- domain-specific data preparation
- domain-specific prompt semantics

### 5.4 Codec

A **codec** is responsible for:

- encoding structured supervision into training text
- decoding generated text back into structured output
- emitting loss metadata
- validating schema and geometry

### 5.5 Prompt Profile

A **prompt profile** is a named configuration of:

- system prompt
- user prompt or template
- output protocol hints
- task-specific decoding defaults

## 6. Target Framework Layout

The framework MUST preserve the current `core / tasks / domains` split. It MAY add cross-cutting modules only for mixing and registry concerns.

```text
src/vlm_structgen/
├── core/
│   ├── config.py
│   ├── registry.py
│   ├── data/
│   ├── train/
│   ├── eval/
│   ├── infer/
│   └── utils/
├── tasks/
│   ├── grounding/
│   ├── keypoint_sequence/
│   └── joint_structure/
├── domains/
│   └── arrow/
├── mixing/
│   ├── sampler.py
│   ├── batcher.py
│   └── strategy.py
└── prompts/
    ├── registry.py
    └── profiles/
```

Allowed cross-cutting modules:

- `mixing`: multi-task dataset mixing and route-aware sampling
- `registry`: adapter / prompt / codec lookup
- `prompts`: profile versioning and reuse

These modules MUST remain orthogonal to the main three-layer hierarchy.

## 7. Layer Boundary Rules

### 7.1 `core`

`core` is the generic framework layer.

`core` is responsible for:

- configuration loading
- dataset and collator plumbing
- trainer and evaluator orchestration
- inference runners
- route registry dispatch

`core` MUST NOT contain:

- arrow-specific geometry rules
- codec-specific JSON schema rules
- canonical ordering rules
- task prompt text
- dataset sampling policy

### 7.2 `tasks`

`tasks` is the task-semantics layer.

`tasks` is responsible for:

- task semantics
- task adapters
- task-specific loss construction
- task-specific metrics

`tasks` MUST NOT contain:

- domain-specific codec details
- data preparation policy
- sampling policy
- deployment protocol details

### 7.3 `domains`

`domains` is the domain-semantics layer.

`domains` is responsible for:

- domain-specific codecs
- canonical ordering
- data preparation
- prompt profile definitions
- inference-side domain conventions

`domains` MUST NOT contain:

- generic trainer logic
- route mixing policy
- task orchestration logic

### 7.4 `mixing`

`mixing` is the multi-task composition layer.

`mixing` is responsible for:

- multi-task dataset composition
- route-aware sampling
- batch route policy

`mixing` MUST NOT contain:

- codec logic
- JSON schema logic
- prompt semantics
- model architecture logic

### 7.5 Architectural Violations

The following are architectural violations:

- trainer parsing task JSON fields directly
- codec choosing training batch policy
- domain deciding optimizer or sampler behavior
- prompt profiles becoming the source of truth for GT schema
- task adapters hardcoding domain-specific data preparation
- a new intermediate orchestrator layer hiding task logic
- a new codec-like layer that also owns training policy
- a new sampling layer that also embeds prompt semantics
- a new task layer that starts owning domain ordering or data preparation

### 7.6 No Further Semantic Layering

The framework MUST NOT introduce more business-meaning layers beyond:

- `core`
- `tasks`
- `domains`

If a new component is proposed, it MUST satisfy one of the following:

1. It is a reusable cross-cutting concern.
2. It belongs unambiguously to exactly one existing layer.

If neither is true, the component is too abstract and MUST NOT be added.

### 7.7 Dependency Direction and Import Rules

The dependency direction MUST be one-way:

```text
core  -> tasks -> domains
core  -> mixing
core  -> prompts
```

Allowed:

- `core` importing `tasks` through registry entry points
- `tasks` importing `domains` codecs and task support
- `mixing` importing only `core` data abstractions and route metadata
- `prompts` importing only config/schema utilities

Disallowed:

- `domains` importing `core.train` or `core.eval`
- `tasks` importing `core.train.trainer`
- `mixing` importing `domains` codec internals
- circular imports between `core`, `tasks`, `domains`

### 7.8 Interface Ownership

The following interfaces are owned and versioned by layer:

- `core`:
  - `TaskAdapter` protocol shape
  - dataset/collator batch contract
  - trainer/evaluator call conventions
- `tasks`:
  - route-specific adapter behavior
  - task-level loss and metric definitions
- `domains`:
  - codec input/output schema
  - canonical ordering and validation rules
- `mixing`:
  - route sampling policy
  - batch route homogeneity policy

Any layer modifying another layer's owned interface MUST update:

- this spec
- affected tests
- affected docs

## 8. Data Contract

Every training or validation record MUST explicitly declare:

- `sample_id`
- `image_path`
- `image_width`
- `image_height`
- `task_type`
- `domain_type`

Task-specific fields MAY be present, but they MUST be part of the formal schema, not ad hoc fields.

### 8.1 Route Must Be Explicit

The dataset loader MUST never infer task/domain from:

- filename
- directory
- prompt text
- target text

### 8.2 Common Record Semantics

All records MUST eventually map to one of the formal routes.

Examples:

- `grounding/arrow`
- `keypoint_sequence/arrow`

### 8.3 Task-Specific Payload

Examples of task-specific payloads:

- `grounding`: instances with `label + bbox_2d`
- `keypoint_sequence`: single target crop with local `label + bbox_2d`, target output `keypoints_2d`
- `joint_structure`: instances with `label + bbox_2d + keypoints_2d`

## 9. Prompt Registry

The framework MUST provide a prompt registry with the following guarantees:

1. Each route can bind to one prompt profile.
2. Prompt profiles are versioned.
3. Training, validation, and inference SHOULD use the same profile family whenever possible.

### 9.1 Prompt Profile Fields

Minimum fields:

- `profile_id`
- `task_type`
- `domain_type`
- `system_prompt`
- `user_prompt`
- `system_prompt_template` optional
- `user_prompt_template` optional

### 9.2 Prompt Rules

- Prompt text MUST not be the only source of semantics.
- Prompt templates MAY inject runtime conditions, but the codec remains authoritative.
- If a prompt profile changes output schema, the codec MUST change with it.

## 10. Codec Registry

Each task MUST have its own codec.

Required codec behavior:

- `encode(...)`
- `encode_with_loss_meta(...)`
- `decode(...)`
- `decode_with_meta(...)`
- `validate(...)`

The codec is responsible for:

- JSON formatting rules
- canonical ordering
- coordinate quantization
- loss metadata spans
- strict and lenient parsing

### 10.1 Codec Contract

The trainer MUST NOT:

- parse JSON with regex
- infer field spans from raw text
- hardcode task-specific output structure

## 11. Multi-Task Mixing Strategy

The framework MUST support multiple datasets in one run.

### 11.1 Supported First-Version Strategy

The first version MUST use:

- route-aware sampling
- homogeneous batches
- one optimizer
- one backbone

This is the safest first step and the closest practical shape to the current codebase.

### 11.2 Later Strategy

Once the first version is stable, the framework MAY allow:

- weighted interleave
- temperature-based sampling
- route-balanced epoch construction
- mixed-route batches only if the loss path is fully per-sample safe

### 11.3 Sampling Goals

The sampling layer MUST be able to express:

- equal task mixing
- data-proportional mixing
- manual task weights
- task upsampling for low-resource routes

## 12. Batch Semantics

### 12.1 First-Version Rule

Each batch MUST contain exactly one route.

Reason:

- current adapter resolution is route-specific
- current loss construction is route-specific
- current evaluation and logging are easier to reason about with homogeneous batches

### 12.2 Batch Route Validation

If a batch contains mixed `task_type` or `domain_type`, the framework MUST fail fast.

### 12.3 Why This Still Counts as Multi-Task Training

Mixed-task training does not require mixed batches.

It requires:

- multiple tasks in one training run
- shared backbone parameters
- task-aware sampling
- task-specific supervision

## 13. Loss and Supervision

### 13.1 Task-Specific Loss Metadata

Each codec MAY emit structured loss metadata, for example:

- label token spans
- bbox token spans
- keypoint coordinate spans

### 13.2 Loss Flow

The target flow MUST be:

```text
gt_struct -> codec.encode_with_loss_meta() -> target_text + loss_meta
         -> adapter.build_target_token_weights()
         -> collator.loss_weights
         -> trainer loss
```

### 13.3 Trainer Responsibility

The trainer MUST only:

- consume loss tensors
- step the optimizer
- aggregate metrics

The trainer MUST NOT understand:

- label semantics
- bbox semantics
- keypoint semantics
- JSON field parsing

## 14. Evaluation and Best Checkpoint Selection

### 14.1 Per-Task Metrics

Each route MUST have its own primary validation metric.

Examples:

- `grounding/arrow`:
  - `bbox_f1_at_iou50`
  - optionally precision and recall
- `keypoint_sequence/arrow`:
  - `end_to_end_score`
  - optionally point distance statistics
- `joint_structure/arrow`:
  - `end_to_end_score`
  - optionally box and point sub-metrics

### 14.2 Global Validation Metric

The framework MUST expose a global validation score for multi-task runs.

Recommended forms:

- weighted mean of normalized primary metrics
- macro average over routes
- manually configured task-weighted score

### 14.3 Best Model Rule

Best checkpoint selection MUST be driven by one declared metric:

- either a global multi-task score
- or a designated primary task metric

The selected metric MUST be explicit in config.

### 14.4 Role of `eval_loss`

`eval_loss` MUST be kept as an auxiliary monitoring signal only.

It is useful for:

- spotting divergence
- comparing optimization stability
- checking whether the model is still learning something

It MUST NOT be the primary best-model criterion for multi-task structured training.

### 14.5 Best-Model Priorities

Recommended priority order:

1. `multi_task_score` computed from normalized task metrics
2. designated primary task metric, if the run is single-task or business-critical
3. `eval_loss` only as a diagnostic fallback, not the main selector

## 15. Config Model

The framework config MUST support:

- backbone model config
- task routes
- prompt profiles
- codec profiles
- data sources
- mix strategy
- route weights
- optimizer settings
- evaluation metrics
- finetune adapter type

### 15.1 Suggested Config Blocks

```yaml
model:
  name_or_path: models/Qwen3-VL-4B-Instruct

finetune:
  mode: lora
  adapter_type: lora   # or dora, full

mixing:
  strategy: route_interleave
  route_weights:
    grounding/arrow: 1.0
    keypoint_sequence/arrow: 1.0

routes:
  grounding/arrow:
    task_type: grounding
    domain_type: arrow
    prompt_profile: arrow.grounding.v2
    codec_profile: grounding.arrow.v1
  keypoint_sequence/arrow:
    task_type: keypoint_sequence
    domain_type: arrow
    prompt_profile: arrow.keypoint_sequence.v2
    codec_profile: keypoint_sequence.arrow.v1

evaluation:
  primary_metric: val/multi_task_score
  task_metrics:
    grounding/arrow: val/bbox_f1_at_iou50
    keypoint_sequence/arrow: val/end_to_end_score
```

## 16. Migration Plan

### 16.1 Phase 1: Two-Task Mixed Training

Scope:

- `grounding/arrow`
- `keypoint_sequence/arrow`

Goals:

- one shared weight
- route-aware dataset mixing
- homogeneous batches
- per-task validation

### 16.2 Phase 2: Generalize the Registry

Goals:

- add more structured tasks
- add more domains
- keep the same route contract

### 16.3 Phase 3: Optional Batch Relaxation

The homogeneous batch constraint MAY be relaxed only after:

- per-sample route dispatch is implemented
- evaluation can still separate routes cleanly
- the loss path is proven safe

## 17. Refactor Acceptance Gates

A refactor PR is acceptable only if all gates pass:

1. **Boundary gate**
   - No new semantic layer added outside this spec.
   - No forbidden cross-layer imports introduced.
2. **Route gate**
   - `task_type/domain_type` remains explicit and required.
   - Mixed-route batch rejection remains active in v1.
3. **Codec gate**
   - Trainer does not parse task JSON directly.
   - Loss metadata continues to come from codec/adapter path.
4. **Metric gate**
   - Per-task validation metrics are reported.
   - `multi_task_score` is used as primary best-model selector.
   - `eval_loss` remains auxiliary.
5. **Compatibility gate**
   - Existing single-task training configs still run.
   - Existing infer/eval routes keep protocol compatibility.
6. **Documentation gate**
   - `docs/architecture.md`, this spec, and impacted task/domain docs are synchronized.
7. **Test gate**
   - Unit tests for routing, adapter resolution, and batch semantics pass.
   - Regression tests for stage1/stage2 metrics pass on smoke subsets.

## 18. Risks

1. **Task interference**
   - High-frequency tasks may dominate training.

2. **Metric confusion**
   - A single scalar can hide task regressions.

3. **Prompt drift**
   - If prompt profiles and codecs diverge, the model will learn inconsistent protocols.

4. **Route imbalance**
   - Rare tasks may underfit without explicit sampling control.

5. **Over-generic trainer**
   - If the trainer starts parsing task semantics, the framework becomes brittle again.

## 19. Open Questions

These must be decided before the framework is expanded to many tasks:

- What is the default route mixing strategy?
- Should best checkpoint use a macro average or a designated primary metric?
- Should route weights be static or scheduled?
- Should prompt profiles be versioned independently from codec profiles?
- Should we support mixed-route batches in v1, or keep homogeneous batches as a permanent constraint?

## 20. Summary

The intended framework is:

- one Qwen3-VL backbone
- many structured tasks
- explicit task/domain routing
- codec-driven supervision
- prompt profiles as controlled task interfaces
- route-aware data mixing
- per-task validation and explicit best-metric selection

This keeps the system close to LLaMAFactory in training ergonomics, while preserving our key advantage:

- structured JSON generation
- schema-aware codecs
- task/domain semantics that are first-class rather than hidden
